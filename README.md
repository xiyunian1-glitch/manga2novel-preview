# Manga2Novel

Manga2Novel 是一个纯前端部署的漫画转小说工具。你把漫画图片拖进浏览器，配置好模型接口后，它会按剧情顺序完成逐页分析、整书综合、章节写作和可选的全书润色。

在线地址：

- 正式站：https://xiyunian1-glitch.github.io/manga2novel/
- 预览站：https://xiyunian1-glitch.github.io/manga2novel-preview/

## 适合做什么

- 把漫画分镜改写成连续小说
- 为漫改项目快速生成文字初稿
- 对同一批图片测试不同模型、不同文风
- 用结构化中间结果反复修稿，而不是每次都整本重跑

## 当前能力

- 纯前端运行，图片预处理和任务状态都在浏览器内完成
- 支持上传多张图片，也支持直接导入整个文件夹
- 自动压缩图片、保留顺序，并支持拖拽重排
- 支持 `OpenAI / OpenRouter / SiliconFlow / DeepSeek / 任意 OpenAI 兼容接口`
- 支持 `Google Gemini` 原生接口
- 支持多套 API 配置档案
- 支持按阶段切换接口和模型
  - 分析流程：逐页分析、分块综合、整书综合
  - 写作流程：章节写作、全书润色
- 支持两种主流程模式
  - `逐页分析`：逐页分析 -> 分块综合 -> 整书综合 -> 章节写作
  - `直综合写作`：逐页分析 -> 整书综合 -> 写作前全书统稿 -> 章节写作
- 支持忠实转写 / 文学改写两种写作模式
- 支持章节写作时附带场景图，或切换为纯文本写作
- 支持自动重试、暂停、继续、跳过当前项、重试当前项、重置任务
- 支持编辑逐页分析、整书综合、写作前统稿、章节正文和最终润色稿
- 支持实时预览，支持导出 `TXT` 和 `MD`
- API Key 保存在浏览器本地，并通过 `AES-GCM` 加密后写入 `LocalStorage`

## 工作方式

这不是“先上传到本项目后端再处理”的架构，而是浏览器直接请求你配置的模型接口。

典型流程如下：

1. 在浏览器中选择接口类型、填写 API Key、设置模型。
2. 上传漫画图片或整个文件夹。
3. 浏览器先在本地完成图片压缩和预处理。
4. 程序按当前流程模式依次执行逐页分析、综合、写作和润色。
5. 结果会持续保存在当前任务状态中，你可以预览、修改、续跑或导出。

## 快速开始

### 1. 准备内容

- 一个可用的 API Key
- 一组按阅读顺序整理好的漫画图片
- 建议使用 Chrome 或 Edge

### 2. 配置 API

在 `API 配置` 面板中：

- 选择服务预设，或手动填写兼容接口地址
- 输入 API Key
- 选择模型，或点击“获取模型”拉取可用列表
- 保存当前配置

如果你不确定怎么选：

- 图片分析阶段优先用支持视觉输入的模型
- 文本写作阶段可以继续用同一模型，也可以单独切到更擅长写作的模型

### 3. 上传图片

在 `漫画图片` 面板中：

- 点击上传图片，或上传整个文件夹
- 确认排序是否正确
- 如有需要，拖拽调整顺序

顺序非常重要。小说输出的叙事顺序直接依赖图片顺序。

### 4. 选择创作方式

在 `创作设置` 和 `队列配置` 中，最常用的是这些项：

- 文风预设
- `System Prompt`
- `Temperature`
- `忠实转写 / 文学改写`
- `逐页分析 / 直综合写作`
- `每组图片数`
- `章节默认分段数`
- `章节写作附带场景图`
- `全书润色`

### 5. 开始处理

点击顶部的开始按钮后，你可以在：

- `处理进度` 里看当前阶段和失败原因
- `小说预览` 里看章节结果

处理中可以随时：

- 暂停
- 继续
- 跳过当前项
- 重试当前项
- 修改中间结果后继续往下跑

## 两种流程模式

### 逐页分析

适合更稳、更细的流程。

执行顺序：

1. 逐页分析
2. 分块综合
3. 整书综合
4. 章节写作
5. 可选的全书润色

特点：

- 中间结构更完整
- 更适合排查哪一页、哪一块出了问题
- 更适合手动修正后续跑

### 直综合写作

适合希望跳过“分块综合”的场景。

执行顺序：

1. 逐页分析
2. 整书综合
3. 写作前全书统稿
4. 章节写作
5. 可选的全书润色

特点：

- 中间链路更短
- 更依赖整书综合质量
- 章节数量由“章节默认分段数”控制

## 参数说明

### 每组图片数

这个参数控制的是“逐页分析阶段每次发几张图给视觉模型”。

- `0` 表示按当前视觉模型自动选择更稳的批次
- 通常建议在 `2-4` 张之间
- 如果设得太大，遇到上游空回、超长或兼容接口问题时，系统可能自动拆批重试

它不等于章节数量，也不等于最终会写成几段。

### 章节默认分段数

这个参数只在 `直综合写作` 模式下生效。

- 它决定整书综合后默认拆成几段进入章节写作
- `0` 表示自动按页数估算

如果你看到“10 张图分析失败后变成 5 + 5”，那是逐页分析批次在重试时自动拆分，和这里不是一回事。

### 章节写作附带场景图

- 开启：章节写作时会把当前场景图片一起发给模型
- 关闭：章节写作只发送结构化文本、台词和写作前统稿

如果经常遇到空回、上下文过大或兼容接口不稳，关闭这一项通常更稳。

### 自动重试相关

- `最大重试次数` 控制单项失败后最多重试多少次
- `重试间隔` 控制重试等待时间
- `并发请求数` 控制预处理和请求并行度

并发不是越大越好。多数接口在 `2-4` 左右更稳。

## 常见建议

- 先保证图片顺序正确，再调模型和 Prompt
- 图片信息密集时，把“每组图片数”调小
- 连续剧情尽量放在同一个任务里跑
- 视觉阶段优先选支持图片输入的模型
- 如果章节写作经常空回，可以先关掉“章节写作附带场景图”
- 如果你要手工继续润色，导出 `MD` 一般比 `TXT` 更方便

## 隐私与数据流

- 图片预处理发生在本地浏览器
- API Key 保存在本地浏览器，并以加密形式存储
- 真正调用模型时，请求会从浏览器直接发送到你选择的服务商

这意味着：

- 本项目默认没有自建后端帮你中转
- 你的图片和提示词不会先经过本项目服务器
- 但会直接到达你配置的目标接口

如果处理的是敏感内容，请先确认所用服务商的隐私策略和审核策略。

## 构建与部署

构建静态站点：

```bash
npm run build
```

构建后会生成适用于 GitHub Pages 的 `out/` 目录，并自动补上 `.nojekyll`。

仓库已包含 GitHub Pages 工作流：

- [`.github/workflows/deploy.yml`](./.github/workflows/deploy.yml)

推荐部署方式：

1. 把源码推到 GitHub。
2. 在仓库 `Settings > Pages` 中把 Source 设为 `GitHub Actions`。
3. 由工作流自动构建并发布。
